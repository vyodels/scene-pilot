# Task: Resume Collection

Collect an already-available resume or attachment for the current candidate without asking the candidate for anything new.

- Prefer resume evidence that is already visible in the active browser scene: online resume content, attachment cards, preview affordances, download affordances, or attachment metadata discoverable from the page.
- Do not send messages, do not request a resume, and do not mutate the recruiting site except browser-side preview or download actions that are strictly necessary to capture an already-visible resume artifact.
- When the path from visible evidence to local artifact requires multiple browser/HID steps, prefer `delegate_scene_context` and pass a stable scene contract with `browser_target`, `computer_target`, `target_regions`, `action_plan`, and `artifact_expectations`. Keep final screen coordinates and DOM-only details out of the business result.
- Treat those fields as explicit tool-surface payload, not runtime magic. If the page already reveals the target tab, download landing region, expected file kind, or post-execution target window, encode them structurally; if one of them is still unknown, state the missing signal instead of replacing it with page-script prose.
- In web HID scenes, preserve host identity from browser-originated evidence only: active tab URL, tab list URL, snapshot URL, or `browser_target.host/url/tab_id`. This host may be transmitted to HID as `target.host` or `context.host` for execution attribution, learning, and trace isolation; do not invent it and do not write site-specific host logic.
- For HID viewport clicks, use `browser_snapshot` clickable `clickPoint` in viewport/document coordinates plus browser-derived target tab/window/host/context. Do not ask browser MCP or recruit-agent to synthesize trusted screen origin or `viewportInScreen`; VirtualHID resolves target window/content viewport to screen mapping.
- Start with scene inspection. Use `browser_snapshot` to understand the visible page first. When snapshot text is insufficient, use `browser_query_elements`, `browser_get_element`, `browser_debug_dom`, or the relevant `browser_wait_for_*` tool to inspect DOM evidence more precisely.
- A visible attachment card, filename, preview link, or download affordance is only an entry point. It is not a completed result by itself.
- If the goal requires a local resume file, continue past scene confirmation and carry the task execution chain through preview or download until you can verify the local artifact path and file format.
- For browser-originated downloads, use browser MCP download-location evidence such as `browser_locate_download` when available. Treat it as read-only Chrome download-record evidence from background/download state; it may report `in_progress`, `interrupted`, or `complete`, and must not use page JS, mock DOM flags, or fixture URL shortcuts as proof that the file landed locally.
- Keep `browser_locate_download` waits short enough for the MCP stdio caller (`waitMs <= 5000` when you set it); if the first lookup misses, retry an immediate or short-poll lookup with the same source correlation instead of issuing a wait longer than the tool transport timeout.
- When `browser_locate_download` returns `located=true`, `state=complete`, `exists=true`, an actual local path such as `file_path`/`filePath`/Chrome `filename`, `fileName`, `extension` or `mime`, and source correlation against the pre-click href/finalUrl/referrer, that record is valid local path and format evidence for a browser-managed download. Do not block solely because the isolated scene lacks a shell `file` command; if stronger local verification is available, use it, otherwise carry the download record into the structured result for business writeback.
- Before HID clicks a download affordance, preserve browser-derived correlation evidence from the snapshot: the clickable `href` / source URL, `download` attribute or expected filename, source `signature/ref`, current tab, and click-before `startedAfter` timestamp. Pass it structurally as `action_plan.download_source` or `artifact_expectations.download_lookup`; later call `browser_locate_download` with that source URL, filename, and timestamp so repeated downloads cannot be mixed up.
- Do not call `submit_result` or return `completed` immediately after confirming that a candidate or attachment exists. First verify whether the file was actually captured locally; if not, keep executing or return a structured failure/replan request.
- When the goal requires a local file, do not return `completed` unless you can verify a real local artifact path and its format from a trusted local/download record or an explicit local file verifier.
- Treat artifact verification and workspace upload as separate obligations. After path/format verification succeeds, use `attach_resume_artifact` to write the verified artifact into the recruiting workspace before treating the broader task execution as fully closed. Do not guess a follow-up `transition_application` status after `attach_resume_artifact`; only transition when a returned thread/allowed transition or explicit operator instruction gives a valid next state.
- If you can only recover text or page evidence but cannot verify a local file, return a structured failure or replan request that explains the gap clearly.
- When you succeed, submit structured result data that includes candidate facts, resume evidence, local artifact path, file format, and any extracted text or contact snapshot that is already visible. Include `artifact` with `artifact_type`, `file_name`, `file_path`, `format`, `verified`, `source_url`, and `final_url`/`referrer` when available; include `browser_download` with the located download record; and include `business_writeback` with `tool: "attach_resume_artifact"` plus arguments suitable for that tool. Prefer a result shape that a later code-level skill or writeback tool can consume directly. Keep the summary business-level; do not echo DOM refs, tab IDs, signatures, or click traces into main history.
- If the latest conversation clearly shows the candidate has withdrawn or wants a temporary cooldown instead of continuing the process, return a structured `rollback_signal` and stop the resume-collection path.
- If this run reveals a stable reusable pattern, bias future skill distillation toward business actions such as `在线简历获取并归档` or `附件简历下载并入库`, and let the future skill accept the same contract fields plus business payload instead of page-specific mechanics.
- Use only explicit candidate intent for rollback. Do not guess.
