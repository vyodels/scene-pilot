# Agent 产品设计思想与原则规范

## 设计目标与适用范围
本文档定义 recruit-agent 在 Agent 产品层面的长期设计思想与原则，用于约束 Autonomous / Assistant、主程序职责、memory、prompt、tool surface、plugin、skill、审批治理、站点接入、UI/API 表达方式等设计方向。

本文档是“设计思想/原则规范”，不是实施计划、handoff、复盘、memory dump，也不是某一时点实现状态的描述。凡是影响 Agent 产品边界与演进方向的设计决策，都应以本文档为上位原则；若实现与本文档冲突，应优先修正实现，或先更新本文档后再调整实现。

本文档与 [`2026-04-20-autonomous-agent-runtime-constraints.md`](./2026-04-20-autonomous-agent-runtime-constraints.md) 的关系如下：
- 后者约束运行时边界、生命周期、prompt 落点、Global Memory 口径、UI 验收边界等长期规则。
- 本文档定义这些规则背后的产品设计立场、职责分层和演进原则。

## 核心设计立场
### 1. Agent 的智能来自自主涌现，不来自主程序硬编码编排
Agent 的核心价值，是 LLM 在明确目标、约束、工具和上下文之上，自主理解当前世界状态并决定下一步动作。

主程序不得把 Agent 设计成预设好的业务状态机、固定步骤脚本或站点操作手册。凡是“先做 A、再做 B、失败则走 C”的业务判断，默认都应属于 Agent 的决策空间，而不是 core runtime 的硬编码工作流。

### 2. 主程序提供环境与边界，不替 Agent 做业务决策
主程序的职责，是提供 prompts、memory、tool surface、plugins、skills、runtime、审批、持久化、UI/API 以及通用基础设施，让 Agent 在这些边界内自主工作。

主程序不负责替 Agent 选择业务路径，不负责预判每个网站或页面的具体操作顺序，也不负责把某项业务经验预先写成固定程序分支。系统应提供“可被智能使用的环境”，而不是“替智能完成思考的流程”。

### 3. 产品默认围绕业务对象与持续状态展开，而不是围绕 runtime 机械结构展开
Agent 产品的第一叙事单元应是目标、业务对象、进展状态、结构化事实、下一步动作与需要的人类介入，而不是 trace、graph、内部节点名或一次性执行脚本。

对用户可见的产品语言，应优先表达“发生了什么、现在处于什么状态、下一步建议是什么、是否需要人工确认”，而不是把 runtime 内部实现细节当作产品主概念。

Dashboard、workspace summary、timeline 或其它 human-facing UI 读模型，不应各自维护第二份独立的业务动作定义、动作标签目录或场景语义副本。系统应以内置的业务场景定义 / scene template 定义作为统一来源，Agent 运行时返回结构化业务动作，展示层只负责把该结构化动作映射为 UI 标签、顺序、颜色和摘要模板。

同理，UI 也不应跨层混排不同语义对象。`Memory` 视图只能承载 memory 层摘要；`Run` / `Goal` / `Approval` 视图只能承载运行态或治理态信息。不得为了“信息更满”而在记忆页顶部继续展示当前业务动作、活跃 run 标题、阻塞说明或当前任务描述。

### 3.1 UI 只能展示后端事实，不制造业务字段
前端不得在任何维度硬编码或 mock 招聘业务字段值，包括但不限于候选人属性、JD 信息、阶段、状态、数量、评分、渠道、时间线和审批状态。真实业务数据必须来自后端 API、共享契约或明确的用户输入；后端缺失的数据应在 UI 中显示为空、禁用或“待获取”，不得用合成值、示例值、随机值或兼容兜底假装已经存在。

当页面需要的字段后端尚未提供时，应先在 plan / spec / PR 说明中记录并与用户确认字段语义、来源和兼容策略，再新增后端字段、共享契约或兼容映射。不得为了让界面看起来完整而先在前端补业务字段。

业务字段、分页、筛选、搜索、统计、评分、简历结构化、话术模板、头像、时间线与投递记录口径的详细治理规则，以 [`2026-04-29-business-fact-contract-governance.md`](./2026-04-29-business-fact-contract-governance.md) 为准。

### 4. 持续演进优先于一次性脚本
系统应鼓励 Agent 在运行中积累可复用经验，并通过 skill、memory 或经审核的提示词资产沉淀长期价值。

项目不应把“当前能跑通一次”视为设计终点，而应把“是否有利于 Agent 在长期运行中越来越稳定、越来越可复用”作为更高优先级的判断标准。

## 主程序与 Agent 的职责分层
| 层级 | 应负责 | 不应负责 |
| --- | --- | --- |
| 主程序 | 提供 prompts、memory schema、tool surface、plugins、skills、runtime、审批治理、持久化、UI/API、通用 Browser/MCP 桥接与基础设施 | 替 Agent 规划业务顺序、预写站点专属流程、在 core 中硬编码业务判断 |
| Agent | 基于目标、状态、约束和工具自主决定下一步动作；组合工具完成任务；在运行中学习和沉淀经验；在需要时请求人类介入 | 依赖主程序替自己完成业务决策；要求系统内置某站点的专属操作剧本 |
| 治理层 | 定义审批、安全、权限、审计、恢复、暂停、接管等制度边界，并在运行时强制执行 | 把安全性寄托于模型自觉；让高风险动作绕过审批和审计 |

这一定义意味着：当 Agent 表现不足时，优先检查目标表达、prompt 约束、工具边界、memory 质量、skill 学习路径和审批制度，而不是立刻在 core runtime 中补写业务分支。

## 双 Agent 产品模型（Autonomous / Assistant）
### 1. 一个共享能力面的双 Agent 产品
Autonomous 与 Assistant 是两个产品角色，而不是两套彼此割裂的能力系统。

二者默认应共享同一类能力面：tool surface、plugin、skill、memory 机制、prompt 体系、审批治理与持久化基础设施都属于统一产品底座。若需要差异化，应通过装配、上下文范围、权限策略、记忆作用域、触发方式和执行策略体现，而不是通过“某项业务能力只挂给某一个 Agent”的方式体现。

共享招聘能力，例如“同步 JD”“发现候选人”“AI 评分”等，应被建模为统一暴露的共享能力域，而不是某个 Agent 的私有动作目录。

### 2. Autonomous 与 Assistant 的差异，应主要体现在以下维度
- `目标`：Autonomous 面向持续推进的长期目标；Assistant 面向人类发起的即时问题与协作请求。
- `记忆范围`：Autonomous 更强调跨 run 的持续状态与长期推进；Assistant 更强调会话协作与即时上下文。
- `生命周期`：Autonomous 是持续存在、可反复被唤醒的主实例；Assistant 是响应式会话代理。
- `执行策略`：Autonomous 适合长流程、自驱运行、等待审批后继续；Assistant 适合即时反馈、对话流式响应和短回路协作。

上述差异不应被误解为“Assistant 没有某类能力”或“Autonomous 必须拥有专属业务动作表”。产品要区分的是运行方式，而不是能力类别。

### 3. Autonomous 的默认产品模型是“一个长期主实例 + 多个 run 历史”
项目应默认围绕一个长期存在的 Autonomous 主实例演进。新的 goal、续跑、补充输入、审批后恢复、定时唤醒与历史追踪，原则上都属于同一主实例生命周期的一部分。

`run` 的职责是记录某一阶段的执行历史、状态转换、审批等待、恢复点和结果留痕，而不是充当一个新的长期 Autonomous 身份。

除非产品层明确引入多 Autonomous 主体的需求，否则不应因为每个新 goal、每次重试或每个业务主题而频繁创建新的 Autonomous 示例。产品默认模型应始终保持“一个主实例持续存在，多次 run 作为其历史轨迹”。

### 4. Assistant 是协作入口，不是 Autonomous 的降级版
Assistant 的价值，在于帮助用户查询、分析、解释、协助操作和派发任务；它不是“缺少能力的 Autonomous”，也不是 Autonomous 的调试壳。

Assistant 与 Autonomous 可以在统一产品底座上协作，但各自都应以清晰的角色边界对外呈现，避免把两者混成一个没有产品语义的“万能 Agent 面板”。

## Memory / Prompt / Tool / Skill / Plugin 的职责边界
### 1. Memory 的职责：沉淀可复用知识，而不是堆积运行细节
Memory 必须分层，并遵循“什么信息该长期保留、什么信息只属于局部执行上下文”的原则。

- `Global Memory` 只承载业务抽象、稳定偏好、长期约束、可复用启发式和跨 run 有价值的经验。
- `Scoped Memory` 承载 JD、候选人、会话或其它明确作用域内的结构化事实与局部经验。
- `子上下文 / 执行单元上下文` 承载网页执行细节、DOM 观察、页面瞬时状态、临时句柄、局部失败链路和一次性操作痕迹。

网页执行细节不得直接提升为 Global Memory。长期记忆记录的应是“未来可复用的原则”，而不是“某次运行当时页面长什么样、按钮叫什么、打开了几个 tab、走过哪条临时路径”。

同样地，`run / goal / round` 的业务动作实例、最近一次业务状态摘要、当前阻塞原因、当前推进建议、当前外部平台状态等运行态信息，也不应直接提升为 Global Memory。它们的正确承载层是 `run context`、`recent events`、`checkpoints`、`logs`、`workspace summary` 或其它面向 human 的运行态投影。

只有当这些运行态信息被进一步提炼为跨 run 仍成立、未来仍可复用的稳定知识后，才允许进入 Global Memory。换言之，`business snapshot` 不是长期记忆本身；它最多只是运行态视图，不能被误当作长期记忆落库。

### 2. Prompt 的职责：作为自然语言行为约束的事实来源
凡是面向 LLM 的自然语言行为约束、步骤指引、成功标准、角色说明、共享任务描述和场景规范，默认都应由 prompt 资产承载，并落在仓库级资源目录 `.recruit-agent/prompts/` 体系下。

`runtime/`、`services/`、工具描述、UI 文案、API 临时拼接文本和注释中，不应继续扩散新的提示词正文。运行时与服务层应传递结构化上下文，例如状态、约束、预算、可用工具、审批状态、作用域和执行结果，而不是在代码中继续追加零散自然语言规则。

### 3. Tool 的职责：提供清晰的原子能力，不伪装成工作流
Tool 应提供语义清晰、边界明确、便于组合的原子能力，尤其优先提供事实查询型能力和结构化写入能力。

主程序不应把业务流程封装成“大而全”的 pipeline 工具，也不应把“任务如何做”的说明错误地混入工具定义。工具的价值在于提供能力原语，而不是替 Agent 预先决定步骤。

### 4. Skill 的职责：沉淀运行中学到的可复用经验
Skill 是 Agent 在运行中形成的可复用策略资产，用于封装经过实践验证的局部经验、技巧或操作路径。

Skill 不应成为把站点专属流程预写进 core 的替代入口。正确方向是：主程序提供 skill 机制与治理边界，Agent 在运行时学习、创建、调用、迭代 skill，经验证后长期复用。

### 5. Plugin 的职责：挂载通用场景能力，不承载站点专属硬编码
Plugin 用于把场景语义、guard checks、context enrichers、tool routers 和其它可插拔能力挂到系统底座上，使产品可以在不污染 Kernel 的前提下扩展能力域。

Plugin 的职责是表达通用业务语义或产品机制，不是把某个外部网站的专属操作流程写成 core 级一等公民。对单一站点有效的经验，应优先沉淀为 Agent 在运行时形成的 skill 或经批准的学习产物，而不是变成主程序默认内置的站点适配层。

项目级 plugin 资产、配置与元数据，默认应落在 `.recruit-agent/plugins/` 下统一管理；`services/backend/src/recruit_agent/plugins/**` 只保留 backend 可 import 的薄运行时 shell / mount code，用于读取这些资产并完成注册与挂载。这条边界表达的是“plugin 资产收口到 `.recruit-agent`”，不是“所有 plugin 可执行代码都搬到 `.recruit-agent`”。

### 6. 共享场景模板的职责：提示词与规范资产，不是工具项
共享场景模板、任务描述页面、可复用规范页和场景说明，本质上都属于 prompt / 规范资产。

凡是用于描述“这个任务是什么、在什么前提下执行、应满足什么完成标准、有哪些交互边界”的共享内容，都不应被误抽象成工具项、工具名或能力枚举。它们的正确位置是任务描述与规范资产层，而不是工具层。

## 站点接入与平台中性原则
### 1. 主程序必须坚持平台中性
主程序不得在 core runtime、主流程、工具表面或站点桥接层中，把 `zhipin.com`、BOSS 直聘或任何其它招聘站点写成特权对象。

系统可以提供通用招聘语义、通用 Browser/MCP 桥接、通用持久化、通用审批、通用记忆和通用技能机制，但不得预置某站点专属工具名、专属流程分支、专属抓取路径、专属下载链路或其它等价硬编码。

### 2. 站点经验应通过运行时学习沉淀，而不是通过主程序写死
面对外部站点时，主程序应假设 Agent 通过通用工具理解页面、识别目标、执行操作并从结果中沉淀经验。

对某个站点有效的经验，应优先沉淀为运行时 skill、可审核学习产物或更高层的抽象提示词约束；不应把站点变化响应、字段识别兜底、页面专属 fallback 或下载补偿逻辑写进 core。

### 3. 平台中性不等于拒绝积累经验
平台中性的含义，是不把站点经验固化成主程序的硬编码前提；它并不否定 Agent 在长期运行中学习站点经验并复用这些经验。

项目的正确方向是“让 Agent 变得更会使用通用平台能力”，而不是“让主程序替 Agent 记住每个站点的专属流程”。

## 人审治理与安全边界
### 1. 高风险动作的人审必须由系统治理层保证
高风险动作是否需要审批，必须由系统治理层、工具元数据、策略配置和运行时拦截机制保证，而不是依赖模型自觉判断。

Agent 可以提出动作意图，但是否允许直接执行、是否进入审批、审批后如何恢复、是否需要暂停或接管，都必须由系统层强制落实，并留下可审计记录。

### 2. 审批、暂停、接管、恢复都属于产品治理能力
审批不是一条普通工具调用，也不是 prompt 中的一句提醒。它是产品治理能力的一部分，必须具备一致的触发标准、状态管理、审计留痕与恢复语义。

同理，暂停、阻塞、handoff、审批后续跑、失败恢复等能力，都应由治理层定义清楚，并通过 runtime 与 UI/API 统一表达，而不是散落在不同工具或临时脚本里。

### 3. 安全边界优先于“让模型自己尽量小心”
当高风险动作、外部触达、破坏性更新、跨对象影响、导出上传或其它需要审慎处理的操作出现时，系统应优先依赖制度化边界，而不是用更多 prompt 文案劝模型“自己谨慎一点”。

Prompt 可以帮助模型理解风险，但不能替代治理机制本身。

## 外部开发代理的工作边界
### 1. 外部开发代理修改系统，不扮演系统内 Agent
Codex 或其它外部开发代理的职责，是修改仓库中的 prompts、memory schema、tool surface、runtime、UI/API、plugin、skill 机制等系统资产，以影响真正运行中的 Agent。

外部开发代理不应越过边界直接扮演内部 Agent，不应占用内部 Agent 的长期身份、memory 或工具链去“替它执行任务”，也不应把开发时的便捷路径伪装成产品能力。

### 2. 主程序 UI 验收可以使用 Playwright / Chrome DevTools
外部开发代理在验收主程序 UI 时，可以通过 Playwright、Chrome DevTools 或等价 UI 自动化工具操作主程序页面。

`browser-mcp` 在本项目中的定位，是供系统内部 Agent 驱动外部 / mock 招聘网站使用，而不是供开发代理直接控制主程序 UI 的通用浏览器。主程序 UI 验收与外部站点执行链路，必须保持边界分离：开发代理可以用 UI 工具触发 `recruit-agent` 的 goal 和查看状态，但不能用 UI 工具替 Autonomous Agent 操作招聘网站。

### 3. 开发代理与运行中 Agent 的边界不能因为工具方便而被打破
即使某些工具在技术上可以直接操作浏览器、页面或数据，开发代理仍应遵守“改系统、不代跑”的边界。任何会模糊“开发行为”和“产品内 Agent 行为”的捷径，都会损害系统设计的一致性。

## 设计变更原则
### 1. 先问是否在增强 Agent，还是在替 Agent 做决定
每次设计变更都应首先回答一个问题：这次变更是在给 Agent 更多可用的目标、上下文、工具、记忆和治理边界，还是在用代码替 Agent 预先完成业务判断。

前者通常是正确方向，后者通常意味着系统正在向硬编码工作流回退。

### 1.5 不得凭空制造新的产品概念
设计与实现时，默认必须复用项目里已经存在并且已被确认的概念，例如 Agent、run、goal、scene template、memory、approval、skill、workspace summary 等。

不得为了表达方便、自我解释或局部实现收敛，而临时创造新的产品概念、命名层或抽象对象，再把它们反向投射回代码、文档、UI 或对外沟通。

如果确实认为现有概念不足以承载新的设计，必须先与用户确认，再明确说明：
- 为什么现有概念不够用
- 新概念会影响哪些代码、文档、UI/API 口径与团队理解
- 引入后的具体价值，以及不引入会带来的真实问题

在用户明确确认前，不得把新增概念写入规范、实现、命名或对外说明。

### 2. 先改 prompt、tool surface、memory 与治理边界，再考虑改 core runtime
当 Agent 在真实环境里表现不佳时，优先检查：
- 目标与成功标准是否表达清楚
- prompt 是否把行为约束放在了正确位置
- tool surface 是否提供了足够清晰的通用原语
- memory 是否在正确层级沉淀了可复用知识
- skill 学习路径与审批治理是否完整

只有当问题明确属于通用运行时缺陷，例如工具暴露、协议兼容、结构化上下文透传、结果持久化、审批状态机、恢复语义等，才应优先修改 core runtime。

### 3. 任何站点专属补丁进入 core 前，都应先被视为设计异味
如果某项改动只能解决单一站点、单一页面、单一下载链路或单一字段识别问题，应默认把它视为设计异味，并优先考虑把问题收敛到 prompt、skill、学习产物或更高层的抽象资产中。

### 4. 规范文档应记录长期原则，而不是临时现状
新增或修改规范时，应写入稳定、抽象、可复用、能指导未来设计取舍的原则；不要把临时 workaround、一次性事故、当前实现细节或阶段性配置写成产品原则。

### 5. 涉及基础边界的变更，应同步更新索引与上位规范
如果变更触及 Autonomous 生命周期、主 conversation / run 模型、Global Memory 口径、prompt 落点、审批治理、站点接入边界、共享能力暴露方式或外部开发代理边界，应同步检查并更新相关索引和上位规范，避免实现与原则文档长期漂移。
