# 业务事实契约治理规范

## 设计目标与适用范围

本文档定义 Recruit Agent 中“业务事实”从后端事实源、共享契约到桌面端展示的长期治理规则。适用范围包括 JD 管理、投递记录漏斗、投递记录跟进、沟通、评分、简历、状态流转、审批、数据看板以及其它招聘业务 UI/API。

本文档是长期规范，不是实施计划、迁移记录或某一轮完成状态。凡是涉及前后端业务字段、分页、筛选、统计、评分、模板、简历结构化、头像、时间线或审批状态的设计与实现，都应遵守本文档。

本规范由 `2026-04-28 Business Fact Contract Hardening Plan` 提炼而来，并已由用户于 2026-04-29 确认迁移为规范。

## 核心原则

### 1. 前端只能展示业务事实，不能制造业务事实

前端不得硬编码、mock、合成或随机生成任何招聘业务字段值，包括但不限于：

- 投递人 / 候选人属性
- 投递记录字段
- JD 信息
- 阶段、状态与状态机节点
- 数量、KPI、趋势、漏斗统计
- AI 评分、简历评分与评分维度
- 来源渠道、沟通渠道
- 时间线、审批、状态流转与动作记录
- 头像、附件、简历摘要、联系方式

真实业务数据必须来自后端 API、共享契约、持久化事实或明确的用户输入。后端没有提供的数据，前端只能渲染为空、禁用或“待获取 / 暂无”，不得用示例值、静态资源、文本猜测或兼容兜底假装已有数据。

### 2. 缺失字段必须先确认语义与来源

当页面需要展示一个后端尚未提供的字段时，处理顺序必须是：

1. 明确字段属于哪个业务对象：投递记录、投递人 / person、JD、run、approval、resume、communication template 或统计投影。
2. 确认字段语义、来源、权限边界、空值语义与兼容策略。
3. 如字段需要前后端共享，优先沉到 `packages/shared` 或后端 typed schema。
4. 后端补齐后，前端再读取和展示。

不得为了页面完整度先在前端补硬编码字段，再把这类字段反向视为产品事实。

### 3. 投递记录页面必须保持投递记录口径

投递记录、投递人、候选人池、person、application 是不同语义对象。当前面向投递记录的页面不得暴露未被产品确认的候选人池口径。

投递记录页面中的动作、沟通、评分、状态流转和历史记录应绑定 `applicationId` / 投递记录身份；需要展示投递人资料时，只能作为投递记录关联 person 的事实投影展示。不得把候选人池字段、投递记录字段和 JD 字段混成同一个无边界对象。

## 事实来源层级

业务事实来源按以下优先级理解：

1. 后端持久化领域对象与服务端投影。
2. 后端 API schema / Pydantic read model。
3. `packages/shared` 中确实跨前后端共享的 TypeScript 契约。
4. 经后端读取和渲染的项目资产，例如 `.recruit-agent/communication_templates/default.json`。
5. 用户在当前 UI 中尚未提交的本地表单草稿、筛选条件和交互状态。

第 5 类只能影响本地交互，不是业务事实来源。提交后必须经过后端 API 成为事实，才能被其它页面或刷新后的 UI 使用。

## 前端实现约束

### 1. 禁止业务字段兜底

前端 normalize、view model、component 和 CSS 不能引入业务字段兜底值。例如：

- 不能给 JD 趋势补 `+3`、`trendRate`、`trendCount`。
- 不能用状态 label 文本 token 在前端归类“沟通中 / 面试中 / Offer 中”。
- 不能从在线简历文本正则猜年龄、学历、工作年限等结构化事实。
- 不能使用静态 demo 头像伪装真实头像。
- 不能给 AI 评分、简历评分、KPI、状态汇总或时间线制造默认业务值。

允许的前端降级仅限于空态表达，例如 `—`、空字符串、禁用态、“待获取”、“暂无记录”，以及头像加载失败后的首字降级。

### 2. 筛选、搜索、分页应由后端承担业务语义

凡是涉及全量结果集语义的筛选、搜索、排序或分页，必须由后端支持。前端可以持有当前输入值和当前页状态，但不能把“当前页过滤”伪装成全量搜索结果。

分页列表 API 应返回 envelope：

```ts
{
  items: T[];
  total: number;
  limit: number;
  offset: number;
  hasNext: boolean;
}
```

后端内部命名可以使用 snake_case，但对前端序列化输出必须与共享契约一致。前端可以接受兼容 alias，但规范字段以共享契约为准。

### 3. 业务统计必须是服务端投影

漏斗统计、KPI、状态数量、评分聚合、时间线数量等业务统计必须由后端服务或共享投影计算。前端不得读取原始列表后自行用文案或局部状态推导业务统计口径。

如果统计口径需要随时间筛选、岗位筛选或权限变化，筛选条件必须成为后端查询契约的一部分。

### 4. 展示组件应区分“无数据”和“数据为零”

缺失数据与数值 `0` 是不同语义：

- 未获取到评分：显示禁用 / 待获取。
- 评分真实为 `0`：显示 `0`，并可查看对应来源或原因。
- 未返回统计：显示缺失态。
- 统计真实为 `0`：显示 `0`。

前端组件必须保留这种区别，不能用 `value || fallback` 这类逻辑吞掉真实 `0`。

## 后端与共享契约约束

### 1. 共享字段必须有 typed contract

同一字段如果同时被后端和桌面端使用，应优先进入：

- 后端 read model / schema
- `packages/shared` TypeScript contract
- 对应 API normalizer 与契约测试

仅在单端内部使用、且不跨边界传播的 UI 状态，才允许留在前端局部类型中。

### 2. 新增字段要带空值语义

新增字段时必须明确：

- 是否可为空
- 为空表示“未知 / 未获取 / 不适用 / 无权限 / 获取失败”的哪一种
- 是否需要来源、时间戳或置信度
- 是否影响筛选、排序、统计或权限

如果这些语义未确认，字段不能先被前端作为业务事实使用。

### 3. 资产型业务能力必须经后端暴露

话术模板、可配置评分 rubric、结构化解析规则等资产可以存放在 `.recruit-agent/` 下，但前端不得直接拼接业务正文或读取资产替代后端能力。

正确链路是：

1. 资产在项目资源目录中维护。
2. 后端负责读取、校验、渲染或投影。
3. 前端调用后端 API 获取列表、渲染结果或缺失态。

### 4. 简历结构化事实应从后端产生

简历结构化事实应由后端解析服务、LLM/规则混合服务或人工确认结果写入。前端只能读取 typed person/contact 字段或 `resumeSnapshot.structured_facts` 这类后端事实。

前端不得从简历全文、在线摘要或聊天内容中正则推断年龄、学历、工作年限、公司、专业、技能等字段。

## 当前确认的业务决策

- JD 列表接口采用分页 envelope：`items,total,limit,offset,hasNext`。
- JD KPI 的“较昨日”增减、`trendRate`、`trendCount` 暂时不是正式业务字段；UI 不展示假趋势。
- 当前用户头像、JD owner/recruiter typed 字段不是已确认业务字段；如需新增，必须先确认身份模型、来源和权限边界。
- 话术模板基础能力可以先以项目资产 + 后端渲染 API 的形式存在；如需运营可编辑，再设计持久化 `CommunicationTemplate` 领域对象。
- JD 漏斗统计应以后端 state machine / milestone 投影为事实来源；是否按当前筛选时间范围计算，需要作为后续产品口径单独确认。

## 设计变更检查清单

涉及业务字段或 UI 展示的改动，提交前至少自检：

- 字段是否来自后端 API、共享契约或用户提交后的持久化事实？
- 是否有前端硬编码、mock、随机、示例值或静态 demo 资源？
- 缺失数据是否显示为空、禁用或待获取，而不是假值？
- `0` 与缺失态是否被区分？
- 搜索、筛选、分页是否覆盖全量数据语义，而不是只过滤当前页？
- 统计、KPI、评分、状态数量是否由后端投影产生？
- 投递记录页面是否仍绑定投递记录口径，没有暴露未确认的候选人池口径？
- 新字段是否已经确认语义、来源、空值、权限和兼容策略？

如果任一项不满足，应先补后端契约、共享类型、产品确认或 plan 记录，再继续实现。
