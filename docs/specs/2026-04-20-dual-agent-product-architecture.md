# 双 Agent 产品架构与协作规范

## 文档目标与适用范围
本文档定义 recruit-agent 中 Assistant 与 Autonomous 的长期产品架构边界、共享能力底座、隔离要求、状态可观测性与场景级协作模型。

本文档适用于所有影响双 Agent 身份建模、能力装配、配置隔离、UI/API 表达、运行状态可见性、handoff 设计和协作关系的实现与验收工作。

本文档记录的是长期稳定的产品规范，不是某个阶段的页面结构、一次性实施计划、handoff 备忘或 memory dump。若现有实现与本文档冲突，应优先修正实现，或先更新本文档后再变更实现。

## 与现有规范的关系
本文档与 [`2026-04-20-agent-product-design-principles.md`](./2026-04-20-agent-product-design-principles.md) 的关系如下：
- 后者说明为什么产品需要两个 Agent 角色，以及两者共享统一能力底座的原则。
- 本文档进一步约束两个 Agent 在身份、配置、可见性、协作和 UI/API 表达上的长期边界。

本文档与 [`2026-04-20-autonomous-agent-runtime-constraints.md`](./2026-04-20-autonomous-agent-runtime-constraints.md) 的关系如下：
- 后者约束 Autonomous 主生命周期、主 conversation / run 模型及若干运行时边界。
- 本文档补充双 Agent 之间的角色分工、隔离约束、协作方式和产品可观测性要求。

## 核心产品模型
recruit-agent 默认是一个共享能力底座上的双 Agent 产品，而不是一个“万能 Agent 面板”附带几种模式切换。

Assistant 与 Autonomous 是两个真正独立的 Agent 身份，拥有不同的目标形态、触发方式、生命周期与协作节奏；但二者应建立在统一的能力底座之上，包括 tool surface、plugin、skill 机制、审批治理、持久化和 prompt 资产体系。

产品需要同时成立两件事：
- 两个 Agent 是独立角色，不能混成一个共享执行上下文的 God Agent。
- 两个 Agent 不是两套彼此割裂的能力系统，不能把共享招聘能力建模成只挂给其中一方的私有动作目录。

## 共享能力底座与差异化装配
### 1. 共享的是能力域，不是执行身份
“同步 JD”“发现候选人”“AI 评分”等招聘业务能力，默认应通过统一的 `plugin / toolkit / MCP / tool surface` 暴露给系统。

这些能力属于共享能力域，不应被包装成“Assistant 专属目录”或“Autonomous 私有挂载层”。

### 2. 差异应通过装配与治理表达
Assistant 与 Autonomous 的差异，默认应通过以下维度表达：
- prompt 与角色说明
- LLM provider / model / budget
- memory 作用域
- tool 权限边界与默认装配
- 生命周期与唤醒方式
- 审批、暂停、恢复和 handoff 策略
- UI/API 的主入口与状态表达

不应通过“是否拥有某项共享招聘能力”来定义两个 Agent 的差异。

## Agent 角色定义
### 1. Assistant
Assistant 是响应式协作入口，面向 human 发起的即时请求。

它的核心职责是查询、解释、分析、辅助操作、派发任务和协作式决策支持。默认情况下，Assistant 应优先呈现短回路、低延迟、强交互的协作体验，而不是承载长时间自驱流程。

### 2. Autonomous
Autonomous 是目标驱动、自主推进的长期执行角色。

它面向持续目标、可被调度唤醒、可跨 run 留存状态，并能够在审批通过后续跑、在受阻时暂停、在适当时发起 handoff。Autonomous 的核心价值，不是回答问题，而是在约束内持续推进任务。

### 3. Assistant 不是 Autonomous 的降级版
Assistant 的定位是协作入口，不是“能力更弱的 Autonomous”；Autonomous 的定位是持续推进器，也不是“会自动点击的 Assistant”。

产品设计必须让用户能够明确分辨两者正在解决的问题类型、交互节奏和责任边界。

## 隔离要求
### 1. 身份与上下文隔离
Assistant 与 Autonomous 不得共享执行身份，不得共用同一段 turn history，也不得把彼此的当前上下文直接混入对方会话。

二者可以共享底层数据与能力基础设施，但运行中的会话上下文、思考轨迹和执行历史必须分离。

### 2. 配置与资产隔离
Assistant 与 Autonomous 应支持独立的：
- system prompt 装配
- model/provider 选择
- token / budget 策略
- skill 资产归属
- 记忆作用域与读写边界

共享的是机制，不是配置实例。

### 3. 审计与历史隔离
两个 Agent 的 run 历史、审批记录、handoff 记录、失败原因和恢复轨迹，都应按 Agent 身份分别可查，不得混在同一条模糊的活动流里。

### 4. UI 主入口隔离
如果产品在同一容器中呈现两个 Agent，仍必须给两者提供清晰分离的主入口、状态视图、配置入口和历史查看路径。

UI 可以共用承载容器，但不能让用户通过一组混杂控件去猜当前自己在与哪个 Agent 交互。

## 状态可观测性要求
### 1. Autonomous 必须是可观测的长期执行体
Autonomous 不得作为黑盒运行。human 与系统必须能实时感知其运行状态、当前处理对象、是否需要人工介入，以及历史发生过什么。

### 2. 最低状态集合
产品和 API 至少应能表达以下状态：
- `idle`
- `running`
- `waiting_human`
- `waiting_handoff`
- `blocked`
- `paused`
- `completed`

如后续扩展更多状态，新增状态也必须保持语义稳定和可审计。

### 3. 进度表达应围绕业务对象
状态展示不应只停留在“正在运行”或“已停止”。对用户可见的表达，默认应包括当前正在处理的目标、JD、候选人、场景节点或下一步动作建议。

产品主语言应围绕业务对象与进展语义，而不是内部 trace、图节点或流程编号。

### 4. 历史与留痕必须持久化
每次运行的状态转换、关键事件、tool call、审批结果、handoff 结果和恢复点，都应可持久化查询，以支持回放、追责与后续接管。

## 场景级 Handoff 规范
### 1. Handoff 的最小单位是业务场景，不是整个 run
Handoff 默认应以场景粒度建模，例如某个候选人的跟进流程、某个 JD 下的一段筛选过程或某个具体沟通节点。

不得把 handoff 简化成“整个 Autonomous run 停止并交给 human”，否则会损失并行推进与局部恢复能力。

### 2. Agent -> Human Handoff
当 Agent 遇到能力边界、连续失败、需要真人沟通或需要情境判断时，可以把当前场景移交给 human。

移交时必须附带可执行的上下文包，至少包括：
- 当前目标
- 已完成内容
- 待完成内容
- 关键数据与对象引用
- 失败原因或移交原因
- 继续推进所需的下一步建议

### 3. Human -> Agent 回交
human 在完成手动处理后，应能把场景明确交回 Agent 继续执行。回交时，Agent 必须从最新业务状态继续，而不是机械重放已完成步骤。

### 4. 其他处理方接管
Handoff 的接收方不应只限定为 human。未来若引入专门的沟通 Agent、谈判 Agent 或其它处理方，接口和数据模型都应保持可扩展，不得把接管方类型写死进主流程。

### 5. Handoff 必须持久化且可查询
场景当前由谁持有、何时移交、为什么移交、是否已回交，都必须作为持久化状态保存，避免重启后丢失责任边界。

## UI / API 表达原则
### 1. 对用户表达角色语义，而不是内部实现结构
产品界面和 API 默认应让用户看到：
- 当前在与哪个 Agent 交互
- 该 Agent 正在处理什么
- 当前状态是什么
- 是否需要 human 介入
- 下一步建议或可执行动作是什么

不应把 runtime 内部节点名、图结构或低层执行器概念作为主产品语言。

### 2. Autonomous 与 Assistant 的入口职责必须清晰
Assistant 入口应强调即时协作、问答与辅助操作；Autonomous 入口应强调目标管理、运行监控、审批/恢复和 handoff 处理。

即使两个入口共用一个界面容器，也应保持任务心智模型清晰，不让用户在一个混合输入框里同时承担两种不同的产品语义。

## 功能演进判断规则
新增 Agent 能力、页面或 API 时，至少应回答以下问题：

1. 这项能力属于共享能力域，还是只是在某个 Agent 上采用不同装配和权限。
2. 这项能力是否会模糊 Assistant 与 Autonomous 的角色边界，导致产品退化成单一 God Agent。
3. 新增状态、审批或 handoff 是否仍能被清晰审计与恢复。
4. 新增 UI 是否让用户更容易判断“当前是谁在做什么”，而不是暴露更多内部实现细节。
5. 若某项能力看似只属于一个 Agent，是否其实应该先抽象成共享能力，再按角色进行装配。

## 变更原则
凡是涉及双 Agent 身份、共享能力暴露方式、配置隔离、状态模型、handoff 粒度、UI 主入口或协作关系的变更，都应先检查本文档是否需要更新。

实现不得先行漂移，再事后补文档。若产品需要偏离本文档约束，应先明确偏离的是哪一类边界，再同步更新本文档与上位规范。
