# 子上下文收口与业务上浮设计草案

**状态**: 草稿
**日期**: 2026-04-20
**背景**: 在 Autonomous UI E2E 主线回归过程中，暴露出“原始观察 / 业务结果 / 长期知识”三层边界尚未真正落地，尤其是浏览器调用过程、候选人业务结果和 skill 沉淀之间的上浮协议还不清晰。本文件仅作为后续优化待办，不在本轮主线中直接实施。

---

## 一、当前待解决问题

### 1.1 子上下文边界尚未清晰

当前系统里，原始浏览器调用过程、工具结果摘要、候选人业务判断和 skill 提炼输入仍有混层风险。需要明确：

- 哪些信息只能留在子上下文局部状态
- 哪些信息允许上浮为父层业务结果
- 哪些信息进一步沉淀为 memory / skill

### 1.2 不能把来源类型 hardcode 到 core runtime

今天的低层观察来源可能是浏览器，明天可能是图片、Excel 或其它 artifact。后续设计不能写成：

- `if browser then ...`
- `if image then ...`
- `if excel then ...`

需要用来源无关的层级协议来约束：

- 原始观察
- 业务结果
- 长期知识

### 1.3 不能把业务判断重新硬编码回主程序

父层不应该通过硬编码规则把 DOM / OCR / Excel 单元格翻译成业务状态；否则仍然是在替 Agent 做业务判断。

---

## 二、当前讨论得到的方向

### 2.1 子上下文是否收口，不由 LLM 单独决定

更合理的职责划分是：

- 系统定义边界与输出协议
- 父层 LLM 判断当前是否需要低层探索
- 子上下文 LLM 在低层探索后产出业务级结果
- 系统只做 schema / 合法性校验，不做业务语义硬编码

### 2.2 父层与子上下文的分工

父层负责：

- 维护当前业务目标、成功标准和已知业务状态
- 判断是否需要低层观察 / 探索 / 取证
- 在子上下文返回后，继续推进业务编排

子上下文负责：

- 使用低层原始工具完成观察、导航、取证、读取 artifact
- 把低层观察归纳为业务级结果
- 不把原始网页 / 图片 / 表格细节直接上浮

### 2.3 “系统决定”应理解为上浮协议，而不是来源枚举

系统不应决定：

- 浏览器要不要收口
- 图片要不要收口
- Excel 要不要收口

系统真正决定的是：

- 父层只接受符合任务 schema 的业务结果
- 原始观察默认只留在 run context / recent events / checkpoints / logs
- memory / skill 只接收进一步抽象后的长期知识

---

## 三、候选方案方向

### 3.1 子上下文输入包的职责

建议后续设计为：

- 系统负责组装可信输入壳
  - 当前 goal / success criteria / run state
  - 当前系统内已知的结构化业务状态
  - 可用业务工具与低层工具清单及 schema
  - 输出协议与边界约束
- 父层 LLM 负责补充本次子任务意图
  - subgoal
  - focus
  - expected_output
  - stop_condition

### 3.2 是否进入子上下文的判定方向

建议后续设计为“父层 LLM 主判定，系统做兜底”：

- 如果当前目标可以直接表达成一个或多个业务工具调用，则不进入子上下文
- 如果为了得到下一步业务动作，必须做低层观察 / 探索 / 取证，则进入子上下文

系统后续可考虑通过工具元数据而不是来源类型做兜底，例如：

- `business_action`
- `observation_raw`
- `artifact_reader`
- `approval`
- `communication`

### 3.3 子上下文输出方向

当前讨论倾向于：子上下文结束时，优先返回业务工具语义，而不是自由文本总结。候选输出类型包括：

- 业务工具调用列表
- blocker
- 极短执行摘要
- evidence refs

这里仍有一个待定问题：

- 子上下文是直接执行业务工具调用，还是先返回“拟调用列表”再由父层统一提交

---

## 四、待后续明确的问题

- 父层如何正式“申请开子上下文”
- 子上下文输入 JSON 的稳定协议长什么样
- 子上下文允许返回的输出 JSON 长什么样
- 子上下文的结束条件由哪些系统规则触发
- 业务工具调用应由子上下文直接提交，还是由父层二次确认后提交
- 浏览器 / 图片 / Excel 等低层来源的证据引用如何统一建模
- skill distill 的输入应如何从 raw tool activity 收敛到业务结果层
