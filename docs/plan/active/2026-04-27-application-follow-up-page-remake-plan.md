# 投递记录跟进页面 1:1 复刻改造计划

## 目标

将当前 `投递记录跟进` 页改造成参考图中的招聘沟通工作台形态：左侧全局窄导航、顶部岗位/时间/日期/状态/搜索/刷新工具条、横向状态流程链、可收起的投递记录侧栏、中间投递记录详情与沟通区、右侧 AI 评分与历史折叠面板。

页面仍以现有系统数据为准，不新增后端契约，不硬编码站点业务逻辑，不删除未确认功能。

## 当前入口与必须保留功能

- 应用入口是 `apps/desktop/src/App.tsx`，实际页面由 `DesktopWorkspace` 承载。
- `投递记录跟进` 对应 `CandidatesKanbanView` 的 `status` tab，核心实现是 `apps/desktop/src/features/status-kanban/StatusKanbanView.tsx`。
- 当前数据来自 `summary.applications`、`applicationThreads`、`stateMachine`、`summary.applicationFollowUpSummaryDefinitions`。
- 必须保留：
  - 岗位筛选、日期筛选、状态/汇总筛选、状态链筛选。
  - 投递记录详情、沟通记录、发送消息、刷新。
  - 人工动作、需要备注的动作、人工修改状态。
  - 在线/线下评分、简历、联系方式、状态历史等可见信息。

## 视觉结构

1. 全局窄导航
   - 改成参考图的白底图标栏样式，保留现有 `home / candidates / settings / agents` 实际入口。
   - 不新增无后端或无现有路由支撑的假功能入口。

2. 顶部工具条
   - 投递跟进页内渲染岗位、时间、开始日期、结束日期、状态视图、投递记录搜索、刷新。
   - `投递记录漏斗 / 投递记录跟进 / JD 管理` 顶部切换能力继续保留在当前 workspace 逻辑中，不作为删除项。

3. 状态流程链
   - 使用现有 `stateMachine.nodes` 与 `ApplicationViewModel.displayStatus` 计数。
   - 复刻图中“节点 label + 数量 + 箭头 + 当前态高亮”的横向密集状态链。

4. 投递记录侧栏
   - 左侧分为“数据看板”和“沟通中列表”两块。
   - 列表来自当前筛选后的投递记录。
   - 增加收起/展开状态：展开为数据看板 + 搜索 + 记录列表；收起为窄头像/状态轨道，仍可切换投递记录。

5. 主沟通区
   - 顶部投递记录卡片展示姓名、状态、基础信息、联系方式、当前/期望岗位、标签。
   - 下方 tabs 保留沟通聊天主视图，并兼容记录、面试安排、评价反馈、备注入口。
   - 消息流继续使用 `communicationLogs`；发送继续走现有 `onCreateEntry`。

6. 右侧详情
   - 三张评分卡：AI 综合评分、在线简历评分、离线简历评分。
   - 评分维度从 `aiScores`、`scorecards`、`assessments`、`matchScore` 派生。
   - 折叠面板用于在线简历、离线简历、历史投递记录、历史状态流转记录。

## 功能适配策略

- 不改后端 API；所有新展示字段都从现有 `ApplicationRecord` / `ApplicationThreadRecord` / `ApplicationViewModel` 派生。
- 没有真实字段的数据使用明确降级显示，例如 `--`、`待评分`、`暂无记录`。
- 页面上的卡片、按钮、折叠项不能只做静态展示：
  - AI 综合评分、在线简历评分、离线简历评分可点击进入评分/简历详情。
  - 在线简历预览、离线简历预览、历史投递记录、历史状态流转记录可展开并可跳转到现有详情抽屉的对应 tab。
  - 投递记录列表、状态链、筛选器、刷新、发送消息、人工动作、人工改状态都要保持真实可用。
- “发送职位”“交换微信”等图中按钮如果没有独立后端动作，不能做假功能；先接到现有消息草稿/发送能力或现有详情/更多入口，独立后端动作待用户确认后再扩展。
- 原 `CandidateTable` 不删除，继续供漏斗页使用；跟进页改为侧栏列表。

## 实施步骤

1. 先抽出可复用 UI primitives：投递人头像、评分卡/评分条、可折叠详情面板、信息卡，统一放在 `kanban-shared`。
2. 写入跟进页专用工作台组装组件，复用现有消息、详情抽屉、人工改状态、状态机动作与上述 primitives。
3. 改 `StatusKanbanView`：从表格 + 弹出沟通面板改为顶部工具条 + 状态链 + 三栏工作台。
4. 补充 CSS：全局窄导航白底复刻、跟进页布局、侧栏收起态、投递记录头部、聊天气泡、右侧评分/折叠面板。
5. 跑 `npm run desktop:typecheck`；如 CSS/TS 类型不通过，修到干净。
6. 启动桌面前端或 Vite 渲染器，做截图级检查；如本环境无法打开浏览器，至少完成构建与说明剩余人工视觉验收点。

## 待确认项

- 当前 `createApplicationEntry` 是否已经代表真实平台外发，还是只落库为沟通记录。改造中先保留原行为和发送入口。
- 图中的“发送职位”“交换微信”是否需要成为独立后端动作。改造中先不新增假功能。

## 二次纠偏要求

基于 2026-04-27 复核反馈，第一轮“大致复刻”不达标，本轮以参考图为准重新收敛：

- 跟进页顶部不再保留 `投递记录漏斗 / 投递记录跟进 / JD 管理` 工作区 tab，页面上方直接进入参考图的筛选工具条。
- `数据看板` 是独立跳转卡，点击进入现有首页数据看板，不在卡片下展示汇总指标。
- 左侧投递记录列表常驻，不再做头像窄栏折叠。
- 状态机链默认只展示主流程；点击展开后展示异常/分支链路。
- 右侧默认只展开 `AI 综合评分`；在线简历评分、离线简历评分、当前沟通岗位、简历预览、历史记录和快捷操作默认收起。
- 聊天窗口不再用大资料卡充当空态，改为对话气泡式投递记录上下文，并继续使用现有沟通记录和发送通道。
- 所有动作继续绑定 `application.id`，不暴露候选人口径。
