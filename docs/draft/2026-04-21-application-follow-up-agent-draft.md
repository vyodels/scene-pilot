# 投递记录跟进独立 Agent 草案

**状态**: 草稿  
**日期**: 2026-04-21  
**背景**: 基于近期对 Autonomous、scene context、CandidateApplication 与投递记录跟进机制的连续讨论，对“是否应为投递记录跟进设置独立 Agent”做阶段性收口。

---

## 一、本文定位

本文只记录当前讨论中已经明确的方向、约束与待决问题。

本文不是正式规范，不覆盖 `docs/specs/` 中已确认的长期规则；后续若要升格为正式规范，需先与现有 Agent 架构草案和运行时约束统一收敛。

---

## 二、已明确的口径

### 2.1 Agent 是通用执行体

- `Agent` 是通用执行体。
- `Assistant`、`Autonomous`、投递记录跟进中的执行单元，本质上都属于 `Agent`。
- “投递记录跟进 Agent”描述的是一种**定义与装配方式**，其差异来自 prompt、上下文、tools、skills、permissions 与 lifecycle，而不是一个新的物种或一套单独的智能体内核。

### 2.2 `subagent` 表达的是委派关系

- `subagent` 更适合表达“某个 Agent 由另一个 Agent 派生或委派出来处理局部任务”的关系。
- 它表达的是运行时关系，不表达本体类型。
- 对外架构设计中，应优先使用统一的 `Agent` 语言；只有在描述委派链路时，才需要使用 `subagent`。

### 2.3 跟进的业务主语是投递记录

- 跟进、沟通、状态推进、阶段判断、独立上下文，默认都应落在**一条投递记录**上。
- `CandidatePerson` 回答的是“这个人是谁”。
- `CandidateApplication` 回答的是“这条投递记录当前推进到了哪里”。
- 因此，真正需要被隔离、被持续推进、被挂接沟通历史和阶段状态的对象，是 application-scoped 的投递记录线程。

### 2.4 业务对象与智能执行体分离

- `CandidateApplication` 是业务真相对象。
- `Agent` 是智能执行体。
- 系统应避免把业务对象直接等同为智能体，也应避免把智能体直接命名成业务对象本身。
- 更合适的方向是：**一条投递记录挂接一个独立的 Agent 上下文实例**，由这个实例在进入该投递记录时加载其专属上下文并推进任务。

### 2.5 “成为投递记录跟进 Agent”依赖定义，不依赖类名

- 是否承担投递记录跟进职责，取决于：
  - 注入了什么 prompt
  - 加载了什么上下文
  - 开放了什么工具和技能
  - 施加了什么边界和权限
- 系统不应再通过新增一个特化类名去表达“这就是投递记录跟进 Agent”。

### 2.6 上下文管理应由投递记录跟进功能负责

- 投递记录专属上下文的装配、持久化、恢复与隔离，应由“投递记录跟进”这一业务功能负责。
- 进入某条投递记录时，应加载该投递记录的：
  - 历史沟通
  - 当前阶段
  - 关联 JD 语境
  - 已知简历 / 联系方式线索
  - 本轮待推进目标
- 离开该投递记录后，这些上下文应继续以投递记录粒度持久化，以便下次恢复。

## 三、当前倾向的目标形态

```text
┌────────────────────────────────────────────────────────────┐
│                        通用 Agent Runtime                  │
│  统一承载：prompt 装配、工具调用、run 执行、权限治理、记忆接口 │
└────────────────────────────────────────────────────────────┘
                              │
               ┌──────────────┴──────────────┐
               │                             │
┌──────────────────────────────┐  ┌──────────────────────────────┐
│      Autonomous 主 Agent      │  │        Assistant Agent       │
│      全局调度与长期目标        │  │        即时协作入口          │
└──────────────────────────────┘  └──────────────────────────────┘
               │
               │ 在需要进入某条投递记录时
               ▼
┌────────────────────────────────────────────────────────────┐
│                  投递记录跟进功能 / Follow-up              │
│  负责按 CandidateApplication 粒度管理专属上下文与恢复逻辑   │
└────────────────────────────────────────────────────────────┘
               │
               ▼
┌────────────────────────────────────────────────────────────┐
│     某条投递记录绑定的 Agent 上下文实例                    │
│  - 对应一条 CandidateApplication                           │
│  - 拥有自己持久化的沟通历史与推进上下文                    │
│  - 每次进入时注入这条投递记录专属上下文                    │
│  - 每次退出后把新结果写回该投递记录线程                    │
└────────────────────────────────────────────────────────────┘
```

这张图表达的重点只有两点：

- `Agent Runtime` 仍应统一。
- 投递记录跟进应以**投递记录粒度的独立 Agent 上下文实例**来承载，而不是把所有候选人线程混进主 Autonomous 的一个大上下文里。

## 四、建议保留的分层

```text
CandidatePerson
  └─ 候选人身份信息

CandidateApplication
  └─ 一条投递记录的业务真相
     ├─ 当前阶段
     ├─ 关联 JD
     ├─ 沟通记录
     ├─ 跟进状态
     └─ 结构化业务结果

Agent
  └─ 通用执行体
     ├─ 由 prompt / context / tools / skills 定义职责
     └─ 在进入某条 CandidateApplication 时加载该投递记录的专属上下文

Run
  └─ 某个 Agent 在某个时段内的一次执行
```

这层分离有几个直接收益：

- `CandidateApplication` 继续作为业务真相落点。
- `Agent` 保持通用，不会因为某个业务场景再裂变出新的核心物种。
- 投递记录跟进上下文拥有清晰隔离边界，不会把多个 JD、多个候选人、多个沟通线程混在一起。

## 五、与当前实现的主要差异

### 5.1 当前实现更像“主 Agent + 临时 scene context”

当前代码已经有：

- `AssistantAgent`
- `AutonomousAgent`
- `AgentKernel`
- `SceneContextService`
- `CandidateApplication` 相关业务模型

但当前更接近：

- 主 Agent 在需要时临时开一个 scene context 做局部处理
- scene context 更偏一次性执行实例
- 投递记录粒度的持久化 Agent 上下文还没有成为明确的一等对象

### 5.2 当前缺的不是业务对象，而是“按投递记录恢复 Agent 上下文”的能力

业务对象侧已经有 `CandidateApplication` 落点。

当前真正缺的，是下列能力的一等建模：

- 某条投递记录自己的持久化上下文
- 进入该投递记录时的上下文恢复
- 离开该投递记录后的上下文写回
- 主 Agent 与投递记录线程之间的切换与回收

## 六、当前更稳的方向

### 6.1 Agent 定义统一

- 保持统一 `Agent Runtime`
- 保持统一 `Agent` 概念
- 不再为“投递记录跟进”单独发明新的核心智能体物种

### 6.2 投递记录上下文独立

- 每条投递记录都应拥有自己的独立上下文
- 该上下文默认应持久化
- 每次使用该 Agent 处理这条投递记录时，都必须注入该投递记录自己的上下文

### 6.3 主 Agent 负责调度，投递记录线程负责专注推进

- 主 `Autonomous` 负责全局目标、优先级、唤醒、队列和跨线程调度
- 某条投递记录进入跟进时，应切到这条投递记录自己的上下文来推进
- 线程内输出应沉回该投递记录的业务字段与沟通历史，不应污染主 Agent 的全局上下文

## 七、待决问题

以下问题当前仍应保持草稿状态，暂不写入正式规范：

### 7.1 UI / API 对外是否暴露“投递记录专属 Agent 实例”

需要进一步确定：

- 对外是否要把它显式展示为一个可见的 Agent 实体
- 还是只把它表达为“该投递记录的独立跟进上下文”

### 7.2 它与 `scene context` 的关系

需要进一步明确：

- `scene context` 是否退化为纯执行隔离层
- 投递记录专属上下文是否成为更高一层、可跨多次执行恢复的持久化上下文

### 7.3 它与 `run` 的关系

需要进一步明确：

- 一次进入某条投递记录的推进，是否等价于一个独立 `run`
- 或者它只是主 Agent 生命周期下的一个 application-scoped 执行片段

### 7.4 它与技能蒸馏的关系

需要进一步明确：

- 线程内细节如何只在局部上下文中消化
- 业务级结论如何在不泄露网页细节的前提下沉淀为 skill 或长期知识

---

## 八、当前建议

- 后续设计讨论中，优先使用“投递记录跟进”“application-scoped 线程”“投递记录专属上下文”这些表达。
- 在没有进一步确认前，先避免继续把该能力命名成新的 Agent 物种或新的领域对象。
- 后续若要进入实现设计，优先讨论：
  - 投递记录专属上下文的数据模型
  - 与 `scene context`、`run`、`AgentSession` 的关系
  - 主 Agent 如何进入、挂起、恢复某条投递记录线程
